# Dynamic Programming: Optimal Substructure and Overlapping Subproblems

## 1. Definition and Concept Overview

Dynamic programming (DP) is an optimization technique applied to problems exhibiting two properties:

1. **Overlapping Subproblems**: The problem can be broken into subproblems which are reused multiple times.
2. **Optimal Substructure**: An optimal solution to the problem contains optimal solutions to its subproblems.

The technique trades memory for time: subproblem results are computed once and stored (memoization or tabulation), eliminating redundant computation. This contrasts with pure recursion, which recomputes overlapping subproblems independently.

---

## 2. Core Principles and Internal Mechanics

### 2.1 Overlapping Subproblems via Itinerary Example

Consider a graph with nodes A, B, C, D, E. Distances:

| Edge | Distance |
|------|----------|
| A→B  | 4        |
| A→C  | 2        |
| B→D  | 1        |
| C→B  | 1        |
| C→D  | 5        |
| D→E  | 3        |

Compute the shortest path distance from A to E.

**Naive recursion**: Evaluate all paths. Two distinct paths to D:
- A → B → D: cost 5
- A → C → B → D: cost 2 + 1 + 1 = 4

Both require computing the distance to D. Without storage, the subtree rooted at D is computed separately for each path that reaches D.

**DP approach**: Compute `dist[node]` = shortest known distance from A to `node`. Process nodes in topological order, updating neighbors. Each node is processed once.

### 2.2 Two Implementation Paradigms

| Approach | Direction | Storage |
|----------|-----------|---------|
| Top-down (memoization) | Recursive, start from target, cache results | Hash map or array for visited states |
| Bottom-up (tabulation) | Iterative, start from base case, fill table | Array indexed by state parameters |

Both achieve the same complexity; choice depends on problem structure and readability.

## 3. Step-by-Step Logical Breakdown

### 3.1 Problem: Shortest Path in Directed Acyclic Graph (DAG)

Input: Graph with nodes 0..n-1, weighted edges, source node s, target node t.

Output: Minimum cost from s to t.

**Step 1: Identify state representation**
State = node identifier. `dp[node]` = minimum cost from source to `node`.

**Step 2: Define base case**
`dp[source] = 0`

**Step 3: Define recurrence**
For each node `u` with outgoing edges `(u, v, weight)`:
```
if dp[u] is known:
    dp[v] = min(dp[v], dp[u] + weight)
```

**Step 4: Determine order of computation**
In a DAG, topological order ensures when processing a node, all incoming paths have been considered. For general graphs with no negative cycles, Bellman-Ford (DP over edges) or Dijkstra (greedy + DP) applies.

**Step 5: Extract solution**
`dp[target]` contains the result if reachable; infinity otherwise.

## 4. Implementation (Python)

```python
from typing import List, Dict, Tuple
from collections import deque

def shortest_path_dag(
    n: int,
    edges: List[Tuple[int, int, int]],
    source: int,
    target: int
) -> int:
    """
    Compute shortest path distance from source to target in a DAG.
    
    Args:
        n: Number of nodes (0..n-1)
        edges: List of (from_node, to_node, weight) tuples
        source: Starting node
        target: Destination node
    
    Returns:
        Minimum distance from source to target, or -1 if unreachable.
    """
    # Build adjacency list
    adj: Dict[int, List[Tuple[int, int]]] = {i: [] for i in range(n)}
    in_degree: List[int] = [0] * n
    
    for u, v, w in edges:
        adj[u].append((v, w))
        in_degree[v] += 1
    
    # Kahn's algorithm for topological order
    topo_order: List[int] = []
    queue = deque([i for i in range(n) if in_degree[i] == 0])
    
    while queue:
        u = queue.popleft()
        topo_order.append(u)
        for v, _ in adj[u]:
            in_degree[v] -= 1
            if in_degree[v] == 0:
                queue.append(v)
    
    # If graph contains cycle, topological sort incomplete
    if len(topo_order) != n:
        raise ValueError("Graph contains a cycle; DAG required")
    
    # Initialize distances: infinity for all nodes except source
    INF = 10**12
    dist = [INF] * n
    dist[source] = 0
    
    # Process nodes in topological order
    for u in topo_order:
        if dist[u] == INF:
            continue
        for v, w in adj[u]:
            # Relaxation step
            candidate = dist[u] + w
            if candidate < dist[v]:
                dist[v] = candidate
    
    return dist[target] if dist[target] != INF else -1
```

### 4.1 Naive Recursive Implementation (For Comparison)

```python
def naive_shortest_path(
    adj: Dict[int, List[Tuple[int, int]]],
    u: int,
    target: int,
    visited: set
) -> float:
    """
    Naive recursive search without memoization.
    Exponential time complexity.
    """
    if u == target:
        return 0
    
    visited.add(u)
    min_dist = float('inf')
    
    for v, w in adj.get(u, []):
        if v not in visited:
            dist = naive_shortest_path(adj, v, target, visited.copy())
            min_dist = min(min_dist, w + dist)
    
    visited.remove(u)
    return min_dist
```

The naive version recomputes paths to `target` from every distinct visitation state. With cycles absent, the number of paths grows factorially in dense graphs.

## 5. Time and Space Complexity Analysis

### 5.1 Top-Down / Bottom-Up DP (DAG Shortest Path)

| Complexity | Value | Explanation |
|------------|-------|-------------|
| Time | O(V + E) | Each node processed once in topological order; each edge relaxed once. |
| Space | O(V + E) | Adjacency list storage + distance array + topological order. |

### 5.2 Naive Recursive

| Complexity | Value | Explanation |
|------------|-------|-------------|
| Time | O(2^(V)) worst-case | In a complete DAG, number of paths grows exponentially. |
| Space | O(V) recursion depth | Call stack depth at most number of nodes. |

**Comparison**:
For V=20, exponential approach may require up to ~1 million recursive calls; DP requires ~20 node visits and ~edge count operations.

## 6. Edge Cases and Failure Scenarios

| Scenario | Handling |
|----------|----------|
| Source equals target | Return 0 immediately; DP initialization handles this when source distance is 0. |
| Target unreachable | Distances remain INF; return sentinel (-1). |
| Graph contains cycle | Topological sort fails; DAG requirement violated. For cyclic graphs with non-negative weights, Dijkstra's algorithm applies; for graphs with negative cycles, problem becomes ill-defined (shortest path not finite). |
| Negative weights without cycles | Topological DP works correctly; Dijkstra fails. |
| Empty graph | Distances remain INF; source unreachable unless source == target. |
| Multiple edges between same nodes | Algorithm processes all; min weight automatically selected via relaxation. |

## 7. Practical Use Cases

- **Sequence alignment** (Needleman–Wunsch, Smith–Waterman): DP over 2D grid for edit distance.
- **Knapsack problems**: Optimization under capacity constraints; DP over items and remaining weight.
- **Shortest path in weighted DAGs**: Critical path analysis in project scheduling.
- **Text justification**: Minimize raggedness by DP over word breaks.
- **Matrix chain multiplication**: Optimal parenthesization to minimize scalar multiplications.

## 8. Limitations and Trade-offs

| Aspect | Consideration |
|--------|---------------|
| State space explosion | DP requires enumerating all distinct states. Problems with high-dimensional states (e.g., multiple constraints) become intractable (curse of dimensionality). |
| Problem structure requirement | Both overlapping subproblems and optimal substructure must hold. Many problems satisfy only one property. |
| Memory consumption | Storing results for all states may exceed memory for large input sizes. Bottom-up DP typically uses a full table; top-down may use less if many states unreachable. |
| Non-trivial order determination | For non-DAG problems, ordering may require additional algorithms (e.g., Bellman-Ford iterations) or additional state dimensions (e.g., bitmask for visited sets). |
| Numerical stability | Large INF values must be chosen to avoid overflow when adding weights. Using `float('inf')` introduces floating-point considerations; integer INF (e.g., 10**15) with explicit checks is preferred. |
