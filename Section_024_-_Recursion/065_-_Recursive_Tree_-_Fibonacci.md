## 1. Definition and Concept Overview

A **recursion tree** is a graphical representation of the sequence of function calls generated by a recursive algorithm. Each node corresponds to a single invocation of the recursive function. The root represents the initial call. Children of a node represent the recursive subcalls made by that invocation. Leaves correspond to base cases where no further recursion occurs.

The recursion tree serves two primary purposes:

- **Visualising the decomposition** of a problem into smaller subproblems.
- **Analysing the complexity** of recursive algorithms by summing the work performed at each level and determining the total number of nodes.

---

## 2. Core Principles and Internal Mechanics

- **Node representation**: Each node stores the input parameters (or a representation thereof) and the computational work performed by that call (excluding work done in subcalls). The work is often denoted as `f(n)` for a call with input size `n`.

- **Tree construction**: When a function makes recursive calls, it spawns child nodes. The shape of the tree depends on how many recursive calls are made and how the input size shrinks.  
  - **Linear recursion** (e.g., factorial) yields a chain (each node has one child).  
  - **Binary recursion** (e.g., Fibonacci) yields a binary tree.  
  - **Multi‑way recursion** (e.g., divide‑and‑conquer with k splits) yields a k‑ary tree.

- **Relationship to call stack**: The recursion tree is a static snapshot of all calls that would be made during execution. In contrast, the call stack only contains the active calls along a single path at any moment. The tree’s depth corresponds to the maximum stack depth.

- **Overlapping subproblems**: In trees where the same input appears in multiple nodes (e.g., Fibonacci), the tree contains redundancy. This indicates that the naive recursion performs repeated work, which can be eliminated via memoisation.

## 3. Step-by-Step Logical Breakdown

### 3.1 Factorial – Linear Recursion Tree

**Definition**:  
`fact(n) = 1` if `n <= 1`  
`fact(n) = n * fact(n-1)` otherwise

**Tree for `fact(4)`**:

```
fact(4)
  |
fact(3)
  |
fact(2)
  |
fact(1)   (base case)
```

- Each node performs a multiplication after its child returns.
- The tree is a simple chain of length `n`. Number of nodes = `n+1` (including the leaf base case). This linear structure indicates time proportional to `n`.

### 3.2 Fibonacci – Binary Recursion Tree with Overlap

**Definition**:  
`fib(0) = 0`, `fib(1) = 1`  
`fib(n) = fib(n-1) + fib(n-2)` for `n >= 2`

**Tree for `fib(4)`**:

```
          fib(4)
         /      \
     fib(3)     fib(2)
     /    \     /    \
 fib(2) fib(1) fib(1) fib(0)
 /    \
fib(1) fib(0)
```

- Leaves are base cases (`fib(1)` and `fib(0)`).
- The tree is a full binary tree except near the leaves.
- Notice that `fib(2)` appears twice (under `fib(3)` and directly under `fib(4)`). This illustrates overlapping subproblems.

### 3.3 Merge Sort – Divide‑and‑Conquer Tree

**Definition**:  
Merge sort on an array of length `n`:
- If `n <= 1`, return (base case).
- Split into two halves, recursively sort each, then merge.

**Tree for `n = 4`** (indices 0–3):

```
        ms(0,3)
       /       \
   ms(0,1)    ms(2,3)
   /    \     /    \
ms(0,0) ms(1,1) ms(2,2) ms(3,3)
```

- Each internal node performs a merge of its two sorted halves. The merge cost is proportional to the length of the subarray.
- The tree is perfectly balanced (logarithmic depth). The total work summed across all levels gives O(n log n).

## 4. Implementation (Python)

### 4.1 Instrumented Recursion to Build Tree

The following code defines a `Node` class to represent each recursive call. The recursive function is augmented to create a node, recursively build its children, and return the node.

```python
class Node:
    """Represents a node in the recursion tree."""
    def __init__(self, name, params):
        self.name = name          # function name
        self.params = params      # tuple of arguments
        self.children = []        # list of child nodes
        self.value = None         # optional: computed result

def build_fib_tree(n: int) -> Node:
    """
    Recursively compute Fibonacci and build the recursion tree.
    Returns the root Node of the tree.
    """
    node = Node("fib", (n,))

    if n <= 1:
        node.value = n
        return node

    left_child = build_fib_tree(n - 1)
    right_child = build_fib_tree(n - 2)

    node.children = [left_child, right_child]
    node.value = left_child.value + right_child.value
    return node
```

### 4.2 Printing the Tree Structure

A simple recursive function to print the tree with indentation.

```python
def print_tree(node: Node, indent: str = "", is_last: bool = True):
    """Print the recursion tree with ASCII lines."""
    # Choose the correct connector
    connector = "└── " if is_last else "├── "
    print(indent + connector + f"{node.name}{node.params} -> {node.value}")

    # Prepare indentation for children
    new_indent = indent + ("    " if is_last else "│   ")

    for i, child in enumerate(node.children):
        is_last_child = (i == len(node.children) - 1)
        print_tree(child, new_indent, is_last_child)

# Example usage
root = build_fib_tree(5)
print_tree(root)
```

**Output for `fib(5)`**:

```
└── fib(5,) -> 5
    ├── fib(4,) -> 3
    │   ├── fib(3,) -> 2
    │   │   ├── fib(2,) -> 1
    │   │   │   ├── fib(1,) -> 1
    │   │   │   └── fib(0,) -> 0
    │   │   └── fib(1,) -> 1
    │   └── fib(2,) -> 1
    │       ├── fib(1,) -> 1
    │       └── fib(0,) -> 0
    └── fib(3,) -> 2
        ├── fib(2,) -> 1
        │   ├── fib(1,) -> 1
        │   └── fib(0,) -> 0
        └── fib(1,) -> 1
```

- The printed structure visually mirrors the binary tree.
- The indentation uses box‑drawing characters for clarity.

## 5. Time and Space Complexity Analysis

The recursion tree directly informs both time and space complexities:

- **Number of nodes** corresponds to the total number of function calls.
  - For linear recursion (factorial): `O(n)` nodes.
  - For binary recursion without overlap (like many divide‑and‑conquer algorithms): `O(2^depth)` nodes, which often simplifies to `O(n log n)` when work per level is considered.
  - For naive Fibonacci: number of nodes is approximately `φⁿ`, leading to exponential time.

- **Work per node** must be considered. In the recursion tree analysis, time complexity is the sum of work performed at each node. For Fibonacci, each node does constant work (addition), so total time ≈ number of nodes. For merge sort, each internal node does work proportional to the subarray size; summing over levels gives `O(n log n)`.

- **Space complexity** is determined by the maximum depth of the tree (the longest root‑to‑leaf path) because that equals the maximum call stack size.
  - Factorial: depth `n` → O(n) space.
  - Fibonacci: depth `n` → O(n) space (even though total nodes are exponential, the stack never exceeds depth `n` due to depth‑first evaluation).
  - Balanced tree (merge sort): depth log n → O(log n) space.

---

## 6. Edge Cases and Failure Scenarios

- **Missing base case**: The tree would grow indefinitely (no leaves), representing infinite recursion. The visual would never terminate.

- **Exponential tree size**: For large `n` (e.g., `n > 40` for Fibonacci), the tree becomes astronomically large; building or printing it is impractical.

- **Deep recursion**: A linear chain of depth > 1000 (in Python) leads to a `RecursionError` because the call stack exceeds the interpreter limit, even though the tree conceptually exists.

- **Multiple recursive calls with varying parameter sizes**: The tree structure can become irregular, making manual analysis more complex.

## 7. Practical Use Cases

- **Educational tool** to teach recursion and demonstrate the difference between linear, binary, and overlapping recursion.
- **Debugging** complex recursive algorithms by visualising the call hierarchy.
- **Algorithm analysis** for divide‑and‑conquer recurrences (e.g., `T(n) = a T(n/b) + f(n)`). Drawing the tree helps compute total work.
- **Identifying optimisation opportunities** – Overlapping subproblems are visually obvious in a recursion tree, motivating memoisation or dynamic programming.

---

## 8. Limitations and Trade‑offs

- **Scalability**: For inputs beyond a trivial size, the tree becomes too large to draw or store.
- **Abstract representation**: The tree shows calls but not the order of execution (depth‑first vs. breadth‑first) or the call stack state.
- **Not a runtime data structure**: Recursion trees are conceptual; building one during execution defeats the purpose of analysing performance (it adds overhead).
- **Overlap blindness**: The tree visually shows redundancy but does not quantify it; additional analysis is required.

Despite these limitations, the recursion tree remains an indispensable mental model for understanding and designing recursive algorithms.
